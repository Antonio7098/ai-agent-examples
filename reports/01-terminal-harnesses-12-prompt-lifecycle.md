## Executive Summary

All four systems treat prompts as **code** rather than configuration, storing prompt templates as source files that are deployed alongside application code. None of the systems implement formal prompt versioning (semantic versioning or automated version tagging), A/B testing infrastructure, or governance approval workflows. The most sophisticated lifecycle management is found in **OpenHands**, which uses Jinja2 templating with bytecode caching, model-specific prompt variants via conditional includes, a critic-based evaluation system, and skill-based extensibility. **opencode** provides layered system prompt composition with plugin hooks. **aider** uses a class hierarchy pattern for prompt organization with HTTP cache-control headers. **HelloSales** uniquely provides immutable prompt metadata with full telemetry integration for prompt execution tracking.

## Per-Repo Findings

### opencode

Prompts are static `.txt` files imported via `import PROMPT_DEFAULT from "./prompt/default.txt"` (`packages/opencode/src/session/system.ts:6`). Provider-specific prompts are stored in `packages/opencode/src/session/prompt/*.txt` with model-ID-based selection in `system.ts:19-32`. Dynamic assembly occurs via `resolvePromptParts()` (`packages/opencode/src/session/prompt.ts:223-317`) and `LLM.Service.stream()` (`packages/opencode/src/session/llm.ts:102-114`). Prompts are composed in layers: agent prompt → provider prompt → custom system → user system. No formal versioning, rollback, A/B testing, or governance exists; git provides implicit versioning and rollback.

### OpenHands

Uses Jinja2 templating (`openhands/sdk/agent/prompts/system_prompt.j2`) with bytecode caching via `FileSystemBytecodeCache` (`openhands/sdk/context/prompts/prompt.py:58-74`) and LRU caching on template retrieval (`prompt.py:77-78`). Two-stage system prompt construction separates static cached prompts from dynamic per-conversation context via `AgentContext` (`openhands/sdk/context/agent_context.py:37-420`). Model-specific prompt variants use conditional includes in templates (`system_prompt.j2:134-148`). Few-shot examples are dynamically generated by `get_example_for_tools()` based on available tools (`openhands/sdk/llm/mixins/fn_call_examples.py:333-405`). Critic-based evaluation via `CriticBase` interface (`openhands/sdk/critic/base.py:65-83`). Skills have version fields following AgentSkills standard. No rollback, A/B testing, or governance workflow.

### aider

Prompts stored as Python class string attributes in `aider/coders/*_prompts.py` files. Class hierarchy: `CoderPrompts` base (`base_prompts.py:1`) → format-specific subclasses (e.g., `editblock_prompts.py:7`). Dynamic assembly via `format_chat_chunks()` (`base_coder.py:1226-1331`) and template substitution via `fmt_system_prompt()` (`base_coder.py:1174-1224`). Uses HTTP cache-control headers for prompt caching (`chat_chunks.py:28-55`) with cache warming via keepalive pings (`base_coder.py:1340-1394`). Few-shot examples placement controlled by `examples_as_sys_msg` model setting (`models.py:131`). No formal versioning, rollback, A/B testing, or governance.

### HelloSales

Prompts defined as Python modules with explicit versioning via `PromptMetadata` dataclass (`backend/src/hello_sales_backend/platform/llm/prompts.py:11-45`). Version field uses simple string suffixes ("v1", "v7"). Prompt identity is immutable via frozen dataclass. Runtime reference via `EffectivePromptRef` with optional checksum. Dynamic assembly via lambda-based builder pattern (`PromptMessageBuilder = Callable[[str], list[ChatMessage]]`). Full telemetry integration: every prompt execution automatically captured in span attributes (`telemetry.py:742-751`). Database persistence of prompt_id, prompt_version, owner_kind, owner_id in `AgentRunRecord` and `AgentTurnRecord`. No rollback, A/B testing, or governance workflow. No explicit prompt caching layer.

## Cross-Repo Comparison

### Converged Patterns

1. **Prompts as Code**: All four systems store prompts as source code/text files deployed with the application, not in external databases or configuration systems.

2. **Git-based Versioning**: All systems rely on git for prompt version control without additional tooling or semantic versioning schemes.

3. **No Rollback Infrastructure**: None provide runtime rollback mechanisms; all rely on git history or code redeployment for recovery.

4. **No A/B Testing**: No system implements traffic splitting or experiment infrastructure for prompt variants.

5. **No Governance Approval**: Prompts are reviewed as code in standard PR reviews; no specialized approval workflows.

6. **Layered Prompt Assembly**: All systems compose system prompts from multiple fragments assembled at runtime.

### Key Differences

|--------|----------|-----------|-------|------------|
| Templating | Plain text `.txt` files | Jinja2 `.j2` with bytecode cache | Python string `.format()` | Python string concatenation |

### Notable Absences

- **No formal prompt versioning**: Versions are git commits, not semantic versions with changelogs
- **No prompt evaluation harness**: No system measures prompt output quality programmatically
- **No A/B testing**: No traffic splitting or experimentation for prompt variants
- **No rollback UI**: No UI or API to roll back prompts without git operations
- **No governance workflow**: No approval gates for prompt changes

### Risks If Unchanged

1. **No rollback capability**: A problematic prompt change requires code deployment to fix, causing potential downtime
2. **No prompt evaluation**: Cannot measure whether prompt changes improve or degrade output quality
3. **No A/B testing**: Cannot safely validate prompt changes in production with real traffic
4. **No governance**: Prompt changes have same review process as code changes, potentially insufficient for customer-facing agents
5. **No caching**: Each prompt execution rebuilds the full prompt, increasing latency and API costs
6. **No templating**: Prompt components cannot be reused across different prompts without duplication

## Synthesis

### Architectural Takeaways

1. **Prompts as code is universal**: All elite systems treat prompts as code, suggesting this is the correct default for typed, tested applications.

2. **Templating enables reuse**: OpenHands's Jinja2 approach shows that templating enables conditional includes and component reuse that would be difficult with plain text.

3. **Caching is fragmented**: Each system approaches caching differently (bytecode, LRU, HTTP headers, none). There is no consensus on prompt caching strategy.

4. **Observability varies widely**: HelloSales has the most sophisticated automatic observability integration; other systems require manual instrumentation.

5. **No system has full lifecycle management**: The most mature lifecycle (OpenHands) still lacks formal versioning, rollback, A/B testing, and governance despite sophisticated templating and evaluation.

6. **Dynamic assembly patterns converge**: Runtime prompt composition via context objects (OpenHands `AgentContext`, HelloSales `ProfiledAgentContextAssembler`) is a common pattern for injecting dynamic content.

### Standards to Consider for HelloSales

1. **Prompt Metadata Standard**: Adopt HelloSales's `PromptMetadata` pattern as a standard for prompt identity across all agent/worker definitions

2. **Immutable Prompt Contract**: The frozen dataclass pattern ensures prompt identity integrity through async execution

3. **Telemetry Integration**: Automatic span attribute capture should be extended to all prompt executions with version, owner, and checksum

4. **Runtime Context Assembly**: The `AgentContext` / `ProfiledAgentContextAssembler` separation enables context strategy versioning independently from prompts

5. **Critic Evaluation Interface**: If evaluation is needed, the `CriticBase` interface from OpenHands provides a clean extension point

### Open Questions

1. Should HelloSales implement prompt caching, and if so, which strategy (bytecode, LRU, HTTP headers)?
2. What is the appropriate governance workflow for prompt changes given HelloSales handles sensitive sales data?
3. How should few-shot examples be managed when HelloSales has multiple agent types?
4. Should prompts be externalized from Python code to enable non-developer updates?
5. What metrics should be captured to measure prompt effectiveness?
6. How should breaking prompt changes be versioned and communicated?
