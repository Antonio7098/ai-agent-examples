## Summary

opencode implements a comprehensive artifact model centered on git-based snapshots for versioning, structured diffs for change tracking, and a patch-based file modification system. Artifacts are tightly integrated with the session/message model, with snapshots captured at step boundaries and stored as git write-tree hashes. Session diffs are persisted to both SQLite and JSON files on disk, enabling traceability across agent runs.

## Answers to Protocol Questions

### 1. What types of artifacts does the system produce?

- **File modification artifacts**: Unified diffs generated by `createTwoFilesPatch` in `edit.ts` (`src/tool/edit.ts:53,97`)
- **Patch artifacts**: Structured patches parsed by `Patch.parsePatch` with `add/update/delete` hunks (`src/patch/index.ts:189-245`)
- **Snapshot artifacts**: Git write-tree hashes captured at step boundaries via `Snapshot.track()` (`src/snapshot/index.ts:279-302`)
- **Session diff artifacts**: `Snapshot.FileDiff[]` objects stored as JSON and in SQLite (`src/session/session.sql.ts:35`, `src/storage/storage.ts:205-206`)
- **Step marker artifacts**: `StepStartPart` and `StepFinishPart` embedded in message history (`src/session/message-v2.ts:222-246`)

### 2. Are artifacts versioned?

**Yes.** Snapshots are stored as git write-tree hashes in an isolated git repository per project (`src/snapshot/index.ts:76-86`). The snapshot service uses `git write-tree` to capture filesystem state at a point in time, and these hashes serve as version identifiers. The `Snapshot.Patch` type references files by hash (`src/snapshot/index.ts:13-16`).

Evidence of versioning:
- `Snapshot.track()` returns a hash (`src/snapshot/index.ts:297`)
- `Snapshot.patch()` retrieves files modified since a given hash (`src/snapshot/index.ts:304-335`)
- `StepFinishPart` records the snapshot hash after each step (`src/session/message-v2.ts:233`)

### 3. Can artifacts be reviewed before application?

**Partial.** The `apply_patch` tool validates patches before applying but does not support user review before application. The `edit` tool generates diffs and requests permission (`ctx.ask()`) but the diff is shown as part of the permission flow, not as a dedicated review step (`src/tool/apply_patch.ts:200-211`).

The UI component `session-diff.ts` renders diffs for review (`src/components/session-diff.ts:108-119`), but this is for viewing historical diffs, not pre-application review.

No formal approval artifact mechanism exists.

### 4. Are artifacts traceable to specific executions?

**Yes.** Each `StepStartPart` and `StepFinishPart` contains a `snapshot` field with the git hash at that point (`src/session/message-v2.ts:225,233`). The `computeDiff` function traces from `step-start` snapshot to `step-finish` snapshot to compute what changed during that step (`src/session/summary.ts:81-99`).

The `PatchPart` schema includes `hash` and `files` array linking patches to specific snapshots (`src/session/message-v2.ts:89-95`).

Session diffs are stored with the session ID in the filename: `session_diff/{sessionID}.json` (`src/storage/storage.ts:205`).

### 5. How are artifacts stored (filesystem, DB, S3)?

**Hybrid: SQLite + Filesystem JSON + Git objects.**

- **SQLite**: `session.summary_diffs` column stores `Snapshot.FileDiff[]` as JSON (`src/session/session.sql.ts:35`)
- **Filesystem JSON**: Diff files stored at `{data}/storage/session_diff/{sessionID}.json` (`src/storage/storage.ts:205`)
- **Git objects**: Snapshot gitdir at `{data}/snapshot/{projectID}/{worktreeHash}/` stores git trees (`src/snapshot/index.ts:81`)

The `Storage` service provides a key-value interface over the filesystem JSON storage (`src/storage/storage.ts:66-72,309-312`).

### 6. Can artifacts be rolled back?

**Yes.** The `SessionRevert` service provides full revert capability:

1. Captures current snapshot before reverting (`src/session/revert.ts:73`)
2. Restores filesystem to snapshot state via `Snapshot.restore()` (`src/session/revert.ts:74`)
3. Applies accumulated patches in reverse via `Snapshot.revert()` (`src/session/revert.ts:75`)
4. Removes messages/parts after the revert point (`src/session/revert.ts:77-90,103-144`)
5. `unrevert()` restores to pre-revert state (`src/session/revert.ts:93-101`)

## Architectural Decisions

### Git-based Snapshotting
opencode uses an isolated git repository (not the user's repo) to store snapshots as git trees. This provides content-addressable storage, efficient diff computation via `git diff`, and natural support for pruning old snapshots via `git gc`. The snapshot gitdir is separate from the user's `.git` to avoid contamination (`src/snapshot/index.ts:81`).

### Snapshot-at-Step Pattern
Snapshots are captured at step boundaries (before and after tool execution) rather than per-file-edit. This provides a coarse-grained but efficient versioning model. The `computeDiff` function chains from `step-start` to `step-finish` snapshots to calculate the delta for a complete tool execution.

### Dual Diff Storage
Diffs are stored both in SQLite (for quick access via session queries) and as JSON files (for detailed inspection and sharing). This balances query performance with detailed diff fidelity.

### Patch as First-Class Artifact
The `PatchPart` type stores patch data directly in the message part stream, allowing patches to be replayed as part of session history. This enables both forward replay (applying patches from a session) and reverse replay (revert).

## Notable Patterns

### Permission Request with Diff
The `edit.ts` and `apply_patch.ts` tools embed full diff content in the permission request metadata, allowing the UI to display exactly what will change before the user approves (`src/tool/edit.ts:104`, `src/tool/apply_patch.ts:206-210`).

### Compaction-aware Diff Filtering
`filterCompacted()` removes compacted messages when computing diffs, ensuring only the meaningful diff between current state and recent summary is shown (`src/session/message-v2.ts:1013-1064`).

### BOM-aware File Handling
The `Bom` utility handles byte-order marks during file write/read to preserve file encoding integrity across platforms (`src/util/bom.ts`, used in `src/tool/apply_patch.ts:81,115,124,222,227,235,250`).

## Tradeoffs

### Snapshot Granularity
Capturing snapshots only at step boundaries (not per-file-edit) is efficient but means individual file changes within a step are not independently addressable. If a step modifies 10 files, all 10 are part of one snapshot.

### Git Dependency for Snapshots
Using git for snapshotting means snapshot operations require spawning git subprocesses. This has overhead and requires git to be installed, though it leverages git's optimized diff algorithms.

### No Formal Artifact Review Flow
While diffs are shown in permission requests, there is no dedicated artifact review/approval step before applying patches. The permission is for the tool call, not for the artifact itself.

### Revert Destroys Message History
Revert removes messages after the revert point from the database. While a snapshot is captured before revert, the original messages are not preserved—they are deleted via `sync.run(MessageV2.Event.Removed, ...)` (`src/session/revert.ts:123-127`).

## Failure Modes / Edge Cases

### Snapshot Hash Mismatch
If git objects are corrupted or pruned too aggressively, `Snapshot.restore()` may fail to recover files. The revert operation catches errors and logs them but does not retry or fall back (`src/snapshot/index.ts:337-360`).

### Patch Application Failure
If `Patch.deriveNewContentsFromChunks()` fails to find expected lines, it throws an error that propagates to the tool result. The error message includes the expected lines but not the actual file content for debugging (`src/patch/index.ts:394`).

### Large File Blocking
Files over 2MB are tracked in `info/exclude` of the snapshot gitdir rather than in the snapshot itself, preventing them from being reverted (`src/snapshot/index.ts:238-256`).

### Heredoc Patch Parsing
The `stripHeredoc()` function only handles simple `<<EOF` patterns and may fail to extract patches passed via complex shell heredoc syntax (`src/patch/index.ts:180-187`).

## Future Considerations

- **Artifact review workflow**: A dedicated review step before applying patches would enable user approval of specific changes
- **Artifact metadata enrichment**: Currently limited to additions/deletions; could add author, timestamp, or custom metadata
- **Cross-session artifact references**: Artifacts are session-scoped; sharing artifacts across sessions requires going through the share service
- **Snapshot retention policy**: Currently uses `git gc --prune=7.days`; a more sophisticated policy based on artifact importance could reduce storage
