## Summary

Opencode uses an Effect-based async composition model for tool execution. Tools execute sequentially within a session loop, with support for parallel resolution of tool definitions and prompt parts. Cancellation via AbortSignal and Fiber.interrupt, exponential backoff retries for LLM calls, and streaming via AI SDK's streamText.

## Answers to Protocol Questions

1. **Are tools executed sequentially or in parallel?**
   Primarily sequentially within session loop. Parallel for tool definition resolution and prompt parts.

2. **Can tool results be streamed?**
   Yes - Shell tool streams output chunks, LLM uses AI SDK streamText, processor emits text-delta and reasoning-delta events.

3. **How are long-running tools managed?**
   Shell tools: configurable timeout (default 120s) via Effect.raceAll. Subagent tasks: separate session. Output streaming for progressive results.

4. **How are tool failures handled?**
   failToolCall records error status, publishes to event bus, LLM stream errors trigger retry, permission rejection breaks loop.

5. **Are tools cancellable?**
   Yes - AbortSignal in Tool.Context, Fiber.interrupt, Deferred.fail with Cancelled.

6. **Are tool calls retried? With what strategy?**
   LLM API calls: yes. Exponential backoff (2000ms initial, factor 2, max 30s without headers). Respects retry-after headers.

7. **Are there compensating actions for failed tools?**
   No. Snapshot/patch tracks file changes but no rollback on failure.

8. **How are tool side effects tracked?**
   Snapshot system for pre/post state, plugin hooks for before/after events, SessionEvent system for Tool.Success/Failed events.

## Architectural Decisions

- Effect monad for async composition (tool.ts:43)
- AI SDK integration via EffectBridge (prompt.ts:529,575)
- Plan mode launches up to 3 agents in parallel (prompt.ts:456-469)

## Notable Patterns

- Effect.raceAll for timeout handling (shell.ts:513-528)
- Plugin hook system for tool execution lifecycle
- Snapshot/patch for state change tracking

## Tradeoffs

- Sequential tool execution is simpler but less performant for independent tools
- No compensating actions means failures leave partial state

## Failure Modes / Edge Cases

- Context overflow errors non-retryable (retry.ts:69)
- 5xx errors retried, non-retryable provider errors without 5xx not retried

## Implications for `HelloSales/`

- Opencode's Effect-based model with explicit cancellation signals could inform HelloSales worker design
- Snapshot/patch pattern useful for tracking HelloSales tool side effects
- Exponential backoff with header-aware delay calculation could improve HelloSales retry reliability
