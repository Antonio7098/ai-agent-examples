## Summary

Aider uses sequential tool execution with comprehensive error classification. Retry via exponential backoff starting at 125ms capped at 60s. Streaming for LLM responses only. Cancellation via KeyboardInterrupt with double-press exit pattern. Git-based side effect tracking with auto-commit.

## Answers to Protocol Questions

1. **Are tools executed sequentially or in parallel?**
   Sequentially. One tool call at a time via while True loop in send_message.

2. **Can tool results be streamed?**
   Partial. LLM responses stream to stdout, but tool outputs themselves are not streamed.

3. **How are long-running tools managed?**
   Limited. API request timeout default 600s. Shell commands run to completion with no timeout. No explicit long-running tool handling with progress tracking.

4. **How are tool failures handled?**
   Categorized retry logic. Exceptions classified as retryable (APIConnectionError, RateLimitError, Timeout) or non-retryable (AuthenticationError, NotFoundError, ContextWindowExceededError).

5. **Are tools cancellable?**
   Yes. Via KeyboardInterrupt (Ctrl+C). First press warns, second press within 2s exits.

6. **Are tool calls retried? With what strategy?**
   Yes. Exponential backoff starting at 125ms, doubling each retry, capped at 60s.

7. **Are there compensating actions for failed tools?**
   Partial. Git errors caught and displayed. No transactional rollback. Auto-commit after successful edits.

8. **How are tool side effects tracked?**
   Git-based. Edited files tracked in aider_edited_files, auto-committed with context. No formal audit trail.

## Architectural Decisions

- Sequential execution simplifies reasoning about tool effects
- Exception taxonomy for retry vs fail decisions
- Git as source of truth for file modifications

## Notable Patterns

- Double-press KeyboardInterrupt pattern
- Exception-based error categorization
- Auto-commit with aider_edits=True flag

## Tradeoffs

- Sequential execution is safe but blocks on long tools
- No tool-level timeout means stuck commands cannot be cancelled
- Git-based tracking requires git integration

## Failure Modes / Edge Cases

- ContextWindowExceededError sets exhausted flag, breaks loop
- Shell commands run until completion, cannot be cancelled mid-flight
- API timeout may occur before tool completes

## Implications for `HelloSales/`

- Aider's exception taxonomy provides a model for classifying HelloSales tool errors
- The auto-commit pattern could inspire HelloSales session snapshots
- Double-press cancellation is user-friendly but may not suit programmatic use

## Questions / Gaps

- No tool-level timeout for shell commands
- No formal side effect audit trail beyond git
- Streaming limited to LLM responses
