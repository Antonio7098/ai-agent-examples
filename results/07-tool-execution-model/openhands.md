## Summary

OpenHands supports both sequential and parallel tool execution via ParallelToolExecutor with ThreadPoolExecutor. Cancellation via ACP agent's _cancel_inflight_tool_calls and WebSocket stop mechanism. Retry with exponential backoff for workspace APIs and LLM calls. Long-running tools managed via timeouts with process kill.

## Answers to Protocol Questions

1. **Are tools executed sequentially or in parallel?**
   Both supported. Default sequential (tool_concurrency_limit=1), configured for parallel via ThreadPoolExecutor. Resource locks serialize access to same resources.

2. **Can tool results be streamed?**
   Yes. Three streaming mechanisms: LLM tokens via StreamingDeltaEvent, ACP tool progress via ACPToolCallEvent, local commands via stdout/stderr threads.

3. **How are long-running tools managed?**
   Timeout-based with configurable per-resource defaults. Local: proc.kill() on TimeoutExpired. Remote: polling with timeout check, exit code -1. Resource locks: 30s file, 300s terminal/browser.

4. **How are tool failures handled?**
   AgentErrorEvent wraps tool errors, ACP agent emits failed ACPToolCallEvent, agent receives error and can correct/retry. No automatic retry of failed tools except workspace API retries.

5. **Are tools cancellable?**
   Yes. ACP agent _cancel_inflight_tool_calls emits failed status for pending tool calls. Remote conversations support _stop threading event.

6. **Are tool calls retried? With what strategy?**
   Yes. Workspace API: 3 retries, exponential backoff (1s, 2s, 4s, 8s, max 5s). LLM: 5 retries, exponential backoff (8s-64s). Critic API: 3 retries, exponential (1s-8s).

7. **Are there compensating actions for failed tools?**
   No. System does not implement compensating transactions. Stop hooks prevent further execution but do not undo completed actions.

8. **How are tool side effects tracked?**
   Via resource declarations and event history. Tools declare resources via declared_resources(). ResourceLockManager acquires per-resource locks. Full event history maintained for replay/recovery.

## Architectural Decisions

- ThreadPoolExecutor for parallel execution (parallel_executor.py:38-91)
- Resource-based serialization prevents deadlocks via sorted lock acquisition (resource_lock_manager.py:84-113)
- Event-driven architecture with action/observation pairs

## Notable Patterns

- ParallelToolExecutor with concurrency configuration
- ResourceLockManager with type-specific timeouts
- Tenacity-based retry decorators

## Tradeoffs

- Resource locks add overhead but prevent conflicting access
- ThreadPoolExecutor may OOM with many parallel tools

## Failure Modes / Edge Cases

- ResourceLockTimeout if lock acquisition fails (resource_lock_manager.py:100-112)
- Remote workspace polling timeout may miss events

## Implications for `HelloSales/`

- ParallelToolExecutor pattern could improve HelloSales throughput for independent tools
- ResourceLockManager concept useful for protecting shared resources (files, terminals)
- Event history for replay could aid in HelloSales debugging

## Questions / Gaps

- No compensating actions for failed tools
- Resource lock timeouts may be too short for complex operations
