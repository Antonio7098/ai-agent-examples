## Summary

Guardrails uses **mutable state** with a bounded history stack pattern. Execution state is accumulated in-memory during guard runs and optionally persisted to a server. State is not event-sourced; history is append-only with max_length truncation. Pydantic models serve as the serialization boundary.

## Answers to Protocol Questions

1. **Is state immutable or mutable by default?**
   Mutable — `Guard.history` is a `Stack` (list), iterations are pushed, `Outputs` objects modified in-place. Deep copies used when safe mutation needed.

2. **What state is persisted vs ephemeral?**
   Ephemeral: `Guard.history` (in-memory), iteration outputs. Persisted: Via `GuardrailsApiClient` to server, serialized via `model_dump()`.

3. **Can execution be reconstructed from persisted state?**
   Yes — `Call.model_validate()`, `Guard.from_dict()`, `ValidationOutcome.from_guard_history()` enable reconstruction.

4. **How is state versioned or migrated?**
   No explicit versioning — uses Pydantic serialization with max history length constraint.

5. **How is conversational/agent state separated from execution state?**
   Configuration (validators, schema) vs execution state (Call history with Iterations) clearly separated.

6. **What are the serialization boundaries?**
   Pydantic models with `model_dump()`/`model_validate()` — `ArbitraryModel` base class allows arbitrary types.

## Architectural Decisions

- **Bounded history stack**: `max_length` constraint prevents unbounded memory growth (`guard.py:137`)
- **TODO for history sink**: Explicitly notes server persistence is not implemented (`guard.py:142-143`)
- **Separation of config vs execution**: `GuardExecutionOptions` passed separately from `Guard` state

## Notable Patterns

- Stack-based history with automatic truncation
- Pydantic native serialization/deserialization
- Thread-local context via `contextvars`
- Deep copy before mutation in runner

## Tradeoffs

- Mutable history prevents true event replay; only recent iterations accessible
- No snapshot/checkpoint mechanism for long-running conversations
- Server-side persistence requires external service

## Failure Modes / Edge Cases

- History truncation loses earlier execution context
- No rollback capability for completed calls
- Serialization of exceptions relies on string conversion (`call.py:98-111`)

## Implications for `HelloSales/`

Guardrails pattern of bounded history could inform HelloSales session management. The separation of configuration state from execution history is worth adopting. However, HelloSales may need snapshot support for long-running agent tasks that guardrails doesn't provide.

## Questions / Gaps

- How does guardrails handle concurrent access to the same guard instance?
- Is there any migration path when changing validator/schema structure?
- No evidence of transaction-style rollback if early iterations fail.
