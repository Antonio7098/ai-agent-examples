## Summary

NeMo Guardrails uses **mutable dataclass-based state** with explicit serialization boundaries. The v2.x runtime maintains comprehensive state including flow states, context, and event history. State can be serialized to JSON and deserialized to resume execution. Clear separation between global context and flow-local context.

## Answers to Protocol Questions

1. **Is state immutable or mutable by default?**
   Mutable — `@dataclass` decorators without frozen mode. Direct attribute mutation throughout (e.g., `state.flow_states[uid] = flow_state`).

2. **What state is persisted vs ephemeral?**
   Persisted: flow_states, flow_configs, rails_config, actions, context, last_events. Ephemeral: internal_events, outgoing_events, context_updates, callbacks.

3. **Can execution be reconstructed from persisted state?**
   Yes — `json_to_state()` reconstructs full state, callbacks re-registered after deserialization. Limitation: partial function callbacks lost in serialization.

4. **How is state versioned or migrated?**
   Explicit version marker in dict wrapper (`"2.x"`). Two separate Colang versions with distinct state handling.

5. **How is conversational/agent state separated from execution state?**
   Two-tier context: Global (`state.context`) for shared variables, Flow-local (`flow_state.context`) for per-flow variables.

6. **What are the serialization boundaries?**
   JSON serialization with reference tracking for circular refs. State passed as `{"version": "2.x", "state": <json>}` wrapper.

## Architectural Decisions

- **Event-driven runtime**: Single-threaded processing with internal event queue
- **Callback-based state changes**: FlowHead and FlowState support position/status callbacks
- **Two-tier context system**: Global vs flow-local variable scoping
- **Reference-tracked serialization**: Handles circular references in state graph

## Notable Patterns

- Dataclass-based state with `@dataclass_json` for serialization
- Explicit cleanup of ephemeral state each cycle
- Version-wrapped serialization for cross-version compatibility
- Expression evaluation context merges flow and global state

## Tradeoffs

- Ephemeral callbacks cannot survive serialization — limits checkpoint/resume for certain flows
- No automatic checkpointing; state serialized only on demand from `generate_async`/`process_events_async`
- last_events capped at 500; may lose earlier context

## Failure Modes / Edge Cases

- Callbacks lost on deserialization may break flow logic relying on them
- Internal event queue overflow not handled
- Circular reference handling in serialization may fail on novel patterns

## Implications for `HelloSales/`

NeMo's two-tier context model (global vs flow-local) is worth studying for HelloSales session management. The serialization approach with version markers provides a migration path. However, HelloSales may need automatic checkpointing rather than on-demand serialization.

- How does the system handle state migration between Colang versions mid-execution?
