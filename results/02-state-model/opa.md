## Summary

OPA (Open Policy Agent) uses **mutable state** with transaction-based consistency. The storage layer provides atomic transactions over policy modules and data documents. Query evaluation is ephemeral and isolated via transactions. Schema versioning exists but migrations are TODO.

## Answers to Protocol Questions

1. **Is state immutable or mutable by default?**
   Mutable — in-memory store contains mutable `data any` and `policies`. Disk store mutates via transactions.

2. **What state is persisted vs ephemeral?**
   Persisted: data documents, policy modules, bundle manifests, schema metadata, wasm modules. Ephemeral: transaction pending updates, query bindings, saveStack/saveSet/saveSupport, comprehension cache, function mocks.

3. **Can execution be reconstructed from persisted state?**
   Yes — `Truncate` for checkpointing, bundle activation reads from store, compiler reconstructed from stored policies.

4. **How is state versioned or migrated?**
   Schema version defined (currently 1). Migrations TODO. Backwards compatibility validation for partition map.

5. **How is conversational/agent state separated from execution state?**
   Not applicable — OPA is a policy engine, not an agent system. Query state is ephemeral per-evaluation. Conversational state not modeled.

6. **What are the serialization boundaries?**
   Storage layer: transactions over key/value store. EvalContext: in-memory only, not serialized. Policy modules: raw `[]byte`. Updates: `storage.Update` struct with Path, Value, IsPolicy.

## Architectural Decisions

- **Transaction isolation**: Consistent snapshot per query via `NewTransaction`
- **Mutable in-memory store**: No copy-on-write; assumes callers treat read data as read-only
- **Plugin architecture**: Persistent store, mutable compiler, lifecycle-managed plugins
- **Trigger callbacks**: `onCommit` receives `TriggerEvent` with policy/data changes

## Notable Patterns

- Transaction-based consistency for concurrent access
- Ephemeral evaluation state isolated from persistent storage
- Inter-query cache with `Clone()` support
- Policy module raw text storage for compilation on demand

## Tradeoffs

- No automatic schema migrations — TODO stated
- In-memory store not durable without disk backing
- Query evaluation cannot be reconstructed (ephemeral)

## Failure Modes / Edge Cases

- Schema version mismatch causes fatal error on init
- Partition validation may reject backwards-incompatible changes
- Transaction abort on panic may leave partial writes

## Implications for `HelloSales/`

OPA's transaction model could inform HelloSales persistence strategy. The separation of ephemeral query state from persistent storage is a good pattern. However, HelloSales needs conversational state that OPA doesn't model.
