## Summary

Guardrails is a input/output validation library for LLM applications. It does NOT implement agent memory, episodic memory, or retrieval systems. Memory is limited to:

1. **Call history**: In-memory `Stack[Call]` tracking validation iterations within a session
2. **Context variables**: Python `contextvars` for async-safe per-call metadata (guard name, document store, call kwargs)
3. **No persistence**: History is ephemeral; no cross-session memory

## Answers to Protocol Questions

1. **What types of memory does the system support?**
   - Scratchpad/Working Memory: No dedicated scratchpad. ContextVars provide temporary per-call storage.
   - Episodic Memory: No. Only stores call history within a session via `Stack[Call]`.
   - Retrieval Systems (RAG, Vector Search): No built-in retrieval. Optional `Embedding` and `VectorDB` classes exist (`embedding.py`, `vectordb/`) but are NOT used for agent memory.
   - Checkpointing/Durable State: No checkpointing.
   - Execution State: Tracked via `Iteration` class within `Call`.
   - Conversational State: Not managed. User passes messages directly to `Guard.__call__`.
   - Long-term vs Short-term: All memory is short-term (session-scoped at best).

2. **Is memory persistent across sessions?**
   - No. `Guard.history` is an in-memory `Stack[Call]` with no persistence layer. The comment at `guardrails/guard.py:142` states: "TODO: Support a sink for history so that it is not solely held in memory".

3. **How is memory compressed or summarized?**
   - No compression or summarization is performed.

4. **How is memory integrated into LLM context?**
   - Messages are passed directly to the LLM via `Guard.__call__`. The `Call` class tracks inputs/outputs but does not inject history into prompts.

5. **What storage backends are supported?**
   - No storage backends for memory. Only optional `VectorDB` integrations (`faiss.py`) exist for document retrieval, not agent memory.

6. **How is memory retrieval triggered (automatic vs explicit)?**
   - Not applicable - no retrieval mechanism for agent memory.

7. **What memory is shared between agents?**
   - None. Each `Guard` instance maintains its own isolated history stack.

## Architectural Decisions

- **In-memory only**: Guardrails was designed as a validation layer, not an agent framework. Memory was explicitly deprioritized.
- **ContextVars over globals**: Uses Python's async-safe `contextvars` to avoid threading issues.
- **Configurable history depth**: `history_max_length` provides a safeguard against unbounded memory growth.

## Notable Patterns

- `Stack[Call]` with `max_length` is the only "memory" mechanism
- Uses Pydantic models for serialization of call history
- OpenTelemetry tracing for observability

## Tradeoffs

| Aspect | Tradeoff |
|--------|----------|
| No persistence | Simple deployment; no state to manage |
| No cross-session memory | Users must implement their own conversation history management |
| Limited history depth | Prevents memory leaks but loses historical context |

## Failure Modes / Edge Cases

- History loss on process restart
- Unbounded memory if `history_max_length` is not set
- No mechanism to reconstruct conversation state

## Implications for `HelloSales/`

- HelloSales should NOT rely on Guardrails for any agent memory capabilities
- Conversation history must be implemented outside Guardrails (e.g., in a database or message store)
- If Guardrails is used for validation only, this is acceptable; if it serves as the agent framework, memory must be added
