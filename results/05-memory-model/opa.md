## Summary

OPA (Open Policy Agent) is a policy engine, NOT an AI agent framework. It does not implement agent memory, conversation history, or episodic memory. Memory-related aspects are limited to:

1. **In-Memory Storage**: Default storage backend is an in-memory store (`storage/inmem/inmem.go`)
2. **Transactional State**: Uses MVCC-style transactions for consistent policy/data evaluation
3. **Global State**: Policies and data are stored and queried, not agent conversational memory
4. **WASM Memory**: WebAssembly runtime has its own memory model (`internal/wasm/`)

No concept of agent scratchpad, conversational context, or persistent conversation history.

## Answers to Protocol Questions

1. **What types of memory does the system support?**
   - Scratchpad/Working Memory: No. OPA evaluates policies against input data; no agent scratchpad.
   - Episodic Memory: No. OPA has no concept of conversation history.
   - Retrieval Systems (RAG, Vector Search): No built-in retrieval. Can query stored documents but not an agentic retrieval system.
   - Checkpointing/Durable State: Transactions provide consistency; no checkpointing for agent sessions.
   - Execution State: Query evaluation uses `bindings` and `storeDict` for evaluation state (`eval.go:451`).
   - Conversational State: None. OPA is stateless between requests.
   - Long-term vs Short-term Memory: Storage layer is either in-memory (short-term) or pluggable (disk, etc.).

2. **Is memory persistent across sessions?**
   - In-memory store: No. Data lost on restart.
   - Disk storage: Optional via `storage/disk/` package.
   - Policies can be loaded from files, bundle archives, or remote URLs.

3. **How is memory compressed or summarized?**
   - No compression. OPA stores raw policy documents and data.

4. **How is memory integrated into LLM context?**
   - No integration. OPA is not an LLM framework. Policy decisions are based on input documents provided at query time.

5. **What storage backends are supported?**
   - In-memory (`storage/inmem/`)
   - Disk (`storage/disk/`)
   - Custom implementations via `storage.Store` interface

6. **How is memory retrieval triggered (automatic vs explicit)?**
   - Not applicable. OPA is a policy engine, not an agent.

7. **What memory is shared between agents?**
   - OPA does not have agents. Multiple policy evaluation contexts share the same storage if configured.

## Architectural Decisions

- **Policy-centric**: OPA's "memory" is policy/data storage, not agent conversational memory
- **Pluggable storage**: Storage interface allows various backends
- **Transaction isolation**: Each policy evaluation sees a consistent snapshot
- **WASM support**: Separate memory model for WebAssembly execution

## Notable Patterns

- `storage.Store` interface pattern for backend abstraction
- `Transaction` for consistent read/write isolation
- `storeDict` with object pooling for evaluation state
- WASM memory instructions separate from Go runtime memory

## Tradeoffs

| Aspect | Tradeoff |
|--------|----------|
| In-memory default | Fast evaluation; no persistence without config |
| Transaction isolation | Consistent reads; potential contention |
| No agent memory | Out of scope - OPA is a policy engine, not an agent |

## Failure Modes / Edge Cases

- In-memory data lost on restart
- Disk storage has different consistency characteristics
- WASM memory sandbox is isolated from Go runtime memory

## Implications for `HelloSales/`

- OPA should NOT be considered for agent memory implementation
- If OPA is used for authorization decisions, it can receive input context from HelloSales
- OPA's storage pattern could inspire a pluggable memory backend for HelloSales

## Questions / Gaps

- No evidence of conversation memory because it's out of scope
- No episodic memory - this is a policy engine
- OPA is fundamentally misaligned with AI agent memory needs
