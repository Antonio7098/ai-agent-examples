AutoGen implements **explicit hierarchical planning** through the MagenticOneOrchestrator, which maintains a task ledger with facts and plan as first-class state. Planning is fully separated from execution — the orchestrator plans before agents act and re-plans when stalls occur.

## Answers to Protocol Questions

1. **Is planning first-class or emergent?**
   First-class. The `MagenticOneOrchestrator` dedicates distinct model calls to building a task ledger (facts + plan) before agent execution begins (`handle_start`:156-190). The plan is stored as `self._plan` (`:97`) and passed to agents via the `ORCHESTRATOR_TASK_LEDGER_FULL_PROMPT`.

2. **Are plans inspectable and modifiable?**
   Plans are inspectable via `save_state()` which returns `self._plan` (`:231`). Plans can be modified mid-execution via `_update_task_ledger()` which regenerates both facts and plan when stalls are detected (`:451-476`).

3. **Can plans be persisted and resumed?**
   Yes. `MagenticOneOrchestratorState` (`:64-72`) includes task, facts, plan, n_rounds, n_stalls. The `save_state()`/load_state()` methods (`:225-245`) support full checkpoint/restore of orchestrator state.

4. **How is re-planning handled on failure?**
   On each `_orchestrate_step`, if `is_progress_being_made` is False or `is_in_loop` is True, `n_stalls` increments (`:394-399`). When `n_stalls >= self._max_stalls`, `_update_task_ledger()` regenerates facts and plan, then `_reenter_outer_loop()` resets agents and broadcasts updated ledger (`:402-406`).

5. **Is planning separated from execution?**
   Yes. The orchestrator handles planning (ledger management) while participants handle execution. Planning uses model calls directly (`self._model_client.create()`) while execution dispatches to agent participants via `GroupChatRequestPublish` (`:436-440`).

6. **How does planning interact with tool execution?**
   The progress ledger evaluates `is_request_satisfied` and `is_progress_being_made` after each agent response (`:387-406`). Tools execute inside agents; the orchestrator observes chat messages and decides next speaker and instructions. Tools are not called by the orchestrator directly.

7. **What is the granularity of plan steps?**
   Plan steps are not represented as discrete structured objects. The plan is free-form text generated by the model (`self._plan` at `:186`). The progress ledger operates at round granularity, with each round producing one `instruction_or_question` for the next speaker (`:409`).

## Architectural Decisions

- **Outer/Inner Loop Separation**: The orchestrator distinguishes an outer loop (task ledger management) from an inner loop (round-by-round orchestration). The outer loop runs once initially then on stall detection; the inner loop runs every turn.
- **Ledger-Based State**: Rather than storing raw messages as state, the orchestrator distillates them into structured ledger entries (facts, plan) that persist across turns and can be updated.
- **Model-as-Orchestrator**: The orchestrator uses the same model client as agents to make planning decisions. This is a "model-centric" rather than "code-centric" orchestration pattern.
- **Explicit Speaker Selection**: The progress ledger always determines `next_speaker` rather than having agents volunteer or contextually select.

## Notable Patterns

- **Closed-book planning**: Facts are gathered in a closed-book manner (`:164-174`), then plan is created based on the team description and facts (`:176-186`). This contrasts with open-book where the model would search for information during planning.
- **Structured JSON output for progress**: Progress ledger uses Pydantic models (`LedgerEntry` at `autogen_agentchat/teams/_group_chat/_magentic_one/_prompts.py:113-118`) to constrain model output to valid schema.
- **Graceful degradation on JSON parse failure**: The orchestrator retries up to `_max_json_retries` (default 10) when parsing progress ledger, with fallback for models without structured output support (`:316-384`).

## Tradeoffs

- **Plan opacity**: Plans are stored as free-form strings. The orchestrator cannot surgically edit specific plan steps — it regenerates the entire plan on stalls. This makes plan audit harder.
- **Model dependency**: Planning quality depends entirely on the model client's capabilities. No fallback to heuristic planners if the model is unreliable.
- **Single-threaded orchestration**: The orchestrator processes one round at a time, selecting a single next speaker per turn. Concurrent multi-agent action within a turn is not supported.

## Failure Modes / Edge Cases

- **Stall loops**: If the progress ledger consistently returns malformed JSON after max retries, a `ValueError` is raised at `autogen_agentchat/teams/_group_chat/_magentic_one/_magentic_one_orchestrator.py:384`.
- **Invalid speaker selection**: If the model returns a speaker name not in `participant_names`, the orchestrator raises `ValueError` at `:432-434`.
- **Max turns**: If `max_turns` is reached, `_prepare_final_answer` is called with reason "Max rounds reached" (`:304-305`).
- **Single participant team**: The progress ledger is overridden to always select the single agent when the team has only one member (`:341-346`).

## Implications for `HelloSales/`

HelloSales lacks any visible planning infrastructure. The `generic_agent` and `observer_agent` are stateless singletons that receive user input and produce responses — no outer-loop planning, no task ledger, no re-planning on stalls. To reach parity with AutoGen's planning:

1. **Add explicit planning step**: Introduce a planner component that, given a user task, first gathers facts then creates a plan before dispatching to agents.
2. **Persist plan state**: Add `task`, `facts`, `plan` fields to the agent run state model so plans survive across turns.
3. **Add re-planning logic**: Detect stalls (repeat failures, no forward progress) and trigger plan regeneration with updated context.
4. **Separate planning from execution**: Plan using one model path; execute using agent participants, not mixing planning prompts into every agent turn.
