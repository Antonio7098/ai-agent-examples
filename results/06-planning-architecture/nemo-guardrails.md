## Summary

NeMo Guardrails uses **implicit planning via event-driven flow state machine**. Flows are specified in the Colang DSL as declarative state machines that execute based on event matching. There is no explicit planner or query planner - execution proceeds through head advancement in flow state machines. Failures trigger flow abort (or restart for activated flows), not replanning.

## Answers to Protocol Questions

1. **Is planning first-class or emergent?**
   Emergent. Flows are declarative specifications, not plans generated by a planner. The "planning" is implicit in the event-matching and head-advancement logic.

2. **Are plans inspectable and modifiable?**
   Inspectable: yes - `FlowState` tracks heads positions and context. Modifiable: no - once a flow starts executing, its elements cannot be changed. The system supports `Goto` and label-based jumps within a flow, but not dynamic plan modification.

3. **Can plans be persisted and resumed?**
   No. The state machine runs to completion. State serialization support exists (`test_state_serialization.py`) but the system is not designed for arbitrary pause/resume of flow execution.

4. **How is re-planning handled on failure?**
   It is NOT handled via replanning. On failure, the flow aborts. For `@activated` flows, the system automatically restarts by pushing a `StartFlow` event (`statemachine.py:1354-1361`). There is no search for alternative plans.

5. **Is planning separated from execution?**
   No. The `run_to_completion()` function is both the planner (matching events to heads) and executor (advancing heads). There is no separation.

6. **How does planning interact with tool execution?**
   Tool execution is via `send` operations in flows that create UMIM (Universal Model Interaction Markup) events. Actions like `GenerateFlowFromNameAction` use LLM to dynamically generate flow code.

7. **What is the granularity of plan steps?**
   Fine-grained: flow elements include SpecOp (send/match), ForkHead, MergeHeads, Label, Goto, Assignment, Return, Abort, etc. Each element represents a single step in the state machine.

## Architectural Decisions

1. **Flows as state machines**: Colang flows are specified as state machines with labeled elements. The runtime advances "heads" through the flow based on event matching.

2. **Event-driven execution**: External events (user messages, LLM responses, action results) trigger flow advancement. The system is reactive, not proactive.

3. **Head forking for concurrency**: OR conditions in flows expand to ForkHead elements that create parallel heads. MergeHeads coalesces results from forked branches.

4. **No search or planning**: The system does not search the solution space. It follows the declared flow path. If a path fails, the flow aborts (or restarts if activated).

5. **Dynamic flow generation**: When an undefined flow is started, the system can generate flow code via LLM (`llm.co:199-234`) and add it to the runtime before execution.

## Notable Patterns

1. **Flow head concept**: Multiple `FlowHead` instances can exist for a single flow instance, representing parallel execution branches. Each head has a position and status (ACTIVE/INACTIVE/MERGING).

2. **Event matching with scoring**: When an event arrives, all matching heads are scored by match quality, and the highest-scoring heads advance (`statemachine.py:305-352`).

3. **Action conflict resolution**: When multiple heads produce actions for the same event, `_resolve_action_conflicts()` coalesces them (`statemachine.py:743-767`).

4. **Catch labels for error recovery**: Flows can define `catch_pattern_failure_label` to redirect to a recovery point instead of aborting (`statemachine.py:775-786`).

## Tradeoffs

| Aspect | NeMo Approach | Alternative |
|--------|---------------|-------------|
| Planning | Declarative flows, no planner | Explicit planner with search |
| Flexibility | Limited to flow specification | Could handle dynamic plan modification |
| Failure recovery | Abort/restart for activated flows | Could try alternative paths |
| Visibility | FlowState is inspectable | No pre-execution plan visibility |

## Failure Modes / Edge Cases

1. **Flow abort on runtime error**: Uncaught exceptions push a `ColangError` event and abort the flow (`statemachine.py:871-893`).

2. **No replanning**: Failed flows do not search for alternatives - they simply abort or restart.

3. **Head synchronization issues**: Forked heads that never reach MergeHeads will leave the flow in a hung state.

4. **Dynamic generation failures**: If `GenerateFlowFromNameAction` fails, the system may not have a fallback.

## Implications for `HelloSales/`

1. **HelloSales has similar implicit approach**: Both systems use implicit planning via execution models rather than explicit planners. However, HelloSales uses LLM-driven tool calling while NeMo uses event-driven flow state machines.

2. **HelloSales has retry but no replanning**: Like NeMo, HelloSales does not have explicit replanning - it retries the same approach on failure rather than searching for alternatives.

3. **HelloSales could benefit from flow concepts**: NeMo's flow head concept and event matching could inform HelloSales' agent loop if more structured orchestration is desired.

4. **HelloSales lacks hierarchical flow composition**: NeMo's flow hierarchy (parent-child via `start`) is more structured than HelloSales' worker composition via Stageflow.
